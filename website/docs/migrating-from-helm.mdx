## Why bother?

Sherpack isn't trying to replace Helm everywhere — it tries to be **better
than Helm at the things that hurt**, and a near drop-in for the rest. If
you've ever wished any of the below were different, this guide is for you.

## Syntax cheat sheet

Almost every Helm template construct has a one-line equivalent in Sherpack.
The converter does this for you, but knowing the mapping helps you read
and tweak the output.

<div className="migration-syntax-grid">
  <div className="migration-syntax-col">
    <div className="migration-syntax-label">
      <span className="migration-syntax-badge migration-syntax-badge-helm">HELM</span>
      <span>Go templates</span>
    </div>

```handlebars
{{- if .Values.ingress.enabled -}}
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: {{ include "myapp.fullname" . }}
  labels:
    {{- include "myapp.labels" . | nindent 4 }}
spec:
{{- range .Values.ingress.hosts }}
  - host: {{ .host | quote }}
{{- end }}
{{- end }}
```

  </div>

  <div className="migration-syntax-arrow" aria-hidden="true">
    <svg viewBox="0 0 60 24" width="60" height="24">
      <path d="M 0 12 L 50 12 M 42 5 L 50 12 L 42 19"
        stroke="#00d4aa" strokeWidth="2" fill="none" strokeLinecap="round" strokeLinejoin="round" />
    </svg>
  </div>

  <div className="migration-syntax-col">
    <div className="migration-syntax-label">
      <span className="migration-syntax-badge migration-syntax-badge-sherpack">SHERPACK</span>
      <span>Jinja2</span>
    </div>

```jinja
{% if values.ingress.enabled %}
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: {{ release.name }}
  labels:
    {{ labels() | nindent(4) }}
spec:
{% for h in values.ingress.hosts %}
  - host: {{ h.host | quote }}
{% endfor %}
{% endif %}
```

  </div>
</div>

<details>
<summary>More mappings</summary>

| Concept | Helm | Sherpack |
|---|---|---|
| Conditional | `{{- if .Values.x }}…{{- end }}` | `{% if values.x %}…{% endif %}` |
| Loop | `{{- range .Values.list }}` | `{% for x in values.list %}` |
| Pipeline filter | `{{ .Values.x \| upper }}` | `{{ values.x \| upper }}` |
| Indent helper | `{{ … \| nindent 4 }}` | `{{ … \| nindent(4) }}` |
| Define / include | `{{ define "x" }}` + `{{ include "x" . }}` | `{% macro x() %}` + `{{ x() }}` |
| Whitespace control | `{{- … -}}` | `{%- … -%}` |
| Chart data | `.Chart.Name` | `pack.name` |
| Release | `.Release.Name` | `release.name` |
| Values | `.Values.x` | `values.x` |
| Files API | `.Files.Get "config.json"` | `files.Get("config.json")` |
| Lookup | `lookup "v1" "Secret" .Release.Namespace "x"` | `lookup("v1", "Secret", release.namespace, "x")` |

</details>

## The new `lookup()` model

Helm's `lookup` queries the cluster at render time during `install`/`upgrade`
and returns `{}` during `helm template`. Sherpack does **the exact same
thing**, but with a twist: every non-empty result raises a structured
warning so non-determinism isn't silent.

## Gotchas worth reading

<div className="migration-gotcha-list">

  <details className="migration-gotcha">
    <summary>
      <span className="migration-gotcha-icon" aria-hidden="true">⚠</span>
      <code>lookup()</code> warnings need an elevated log level
    </summary>
    <p>
      Sherpack uses <code>tracing</code>. The CLI defaults to <code>warn</code> for
      sherpack crates and prints to stderr — you'll see lookup warnings.
      For more detail, set <code>RUST_LOG=info</code> or pass <code>--debug</code>.
    </p>
  </details>

  <details className="migration-gotcha">
    <summary>
      <span className="migration-gotcha-icon" aria-hidden="true">⚠</span>
      <code>.helmignore</code> isn't honored at packaging
    </summary>
    <p>
      The converter renames it to <code>.sherpackignore</code> but{' '}
      <code>sherpack package</code> doesn't read it yet. If you have files
      you don't want shipped, prune them before packaging.
    </p>
  </details>

  <details className="migration-gotcha">
    <summary>
      <span className="migration-gotcha-icon" aria-hidden="true">⚠</span>
      Some Sprig functions don't have direct Sherpack equivalents
    </summary>
    <p>
      <code>dateModify</code>, <code>dateInZone</code>, <code>htpasswd</code>,
      <code>bcrypt</code>, <code>fromToml</code>/<code>toToml</code>,
      and OS/URL helpers aren't ported. The converter flags these so you
      see them in the warning report.
    </p>
  </details>

  <details className="migration-gotcha">
    <summary>
      <span className="migration-gotcha-icon" aria-hidden="true">⚠</span>
      Plugin ecosystem doesn't exist (yet)
    </summary>
    <p>
      Tools like helm-diff, helm-secrets, helm-git don't have direct
      replacements. Native Sherpack covers diff (<code>--diff</code>),
      idempotent secret generation, and lock files — but if you're
      heavily plugin-dependent, evaluate carefully.
    </p>
  </details>

  <details className="migration-gotcha">
    <summary>
      <span className="migration-gotcha-icon" aria-hidden="true">✓</span>
      You can run Sherpack and Helm side by side
    </summary>
    <p>
      Sherpack stores releases under a different label
      (<code>app.kubernetes.io/managed-by=sherpack</code>) and uses
      different annotations. Helm releases stay invisible to Sherpack
      and vice versa — handy during incremental migration.
    </p>
  </details>

</div>

## FAQ

<details className="migration-faq">
<summary>Will my existing Helm charts still install with Helm?</summary>
<p>
Yes. <code>sherpack convert</code> writes a new Pack into a separate
directory; your original chart is untouched. Run both side by side
until you're confident in the migration.
</p>
</details>

<details className="migration-faq">
<summary>What about subcharts and umbrella charts?</summary>
<p>
Sherpack handles subcharts with the same value-scoping and global-values
semantics as Helm. The <code>charts/</code> directory becomes
<code>packs/</code>; references in <code>Pack.yaml</code> mirror
Helm's <code>Chart.yaml</code> dependency block.
</p>
</details>

<details className="migration-faq">
<summary>Is the rendered output byte-identical to Helm's?</summary>
<p>
For most charts, yes — modulo whitespace and the differences listed in
the cheat sheet. Run <code>sherpack template</code> and{' '}
<code>helm template</code> on the same inputs and diff to verify
before installing in production.
</p>
</details>

<details className="migration-faq">
<summary>Can I use Sherpack with ArgoCD or Flux?</summary>
<p>
Yes — render with <code>sherpack template</code> and let the GitOps
controller apply. <strong>Don't mix imperative <code>sherpack install</code></strong>{' '}
with GitOps reconciliation; pick one.
</p>
</details>

<details className="migration-faq">
<summary>Performance vs Helm?</summary>
<p>
Native Rust binary, ~19 MB, no Go runtime. Render times for typical
charts are sub-second. The MiniJinja engine is fast, but the real
win is search (SQLite FTS5) and dependency resolution.
</p>
</details>

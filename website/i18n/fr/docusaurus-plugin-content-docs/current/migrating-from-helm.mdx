## Pourquoi se donner la peine ?

Sherpack n'essaie pas de remplacer Helm partout — il essaie d'être **meilleur
que Helm sur ce qui fait mal**, et un quasi drop-in pour le reste. Si tu
as déjà souhaité que l'un des points ci-dessous soit différent, ce guide
est pour toi.

## Antisèche de syntaxe

Quasiment chaque construction de template Helm a un équivalent en une
ligne dans Sherpack. Le converter le fait pour toi, mais connaître la
correspondance aide à lire et ajuster le résultat.

<div className="migration-syntax-grid">
  <div className="migration-syntax-col">
    <div className="migration-syntax-label">
      <span className="migration-syntax-badge migration-syntax-badge-helm">HELM</span>
      <span>Go templates</span>
    </div>

```handlebars
{{- if .Values.ingress.enabled -}}
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: {{ include "myapp.fullname" . }}
  labels:
    {{- include "myapp.labels" . | nindent 4 }}
spec:
{{- range .Values.ingress.hosts }}
  - host: {{ .host | quote }}
{{- end }}
{{- end }}
```

<details>
<summary>Plus de correspondances</summary>

| Concept | Helm | Sherpack |
|---|---|---|
| Conditionnel | `{{- if .Values.x }}…{{- end }}` | `{% if values.x %}…{% endif %}` |
| Boucle | `{{- range .Values.list }}` | `{% for x in values.list %}` |
| Filtre pipeline | `{{ .Values.x \| upper }}` | `{{ values.x \| upper }}` |
| Helper indent | `{{ … \| nindent 4 }}` | `{{ … \| nindent(4) }}` |
| Define / include | `{{ define "x" }}` + `{{ include "x" . }}` | `{% macro x() %}` + `{{ x() }}` |
| Whitespace | `{{- … -}}` | `{%- … -%}` |
| Données chart | `.Chart.Name` | `pack.name` |
| Release | `.Release.Name` | `release.name` |
| Values | `.Values.x` | `values.x` |
| API Files | `.Files.Get "config.json"` | `files.Get("config.json")` |
| Lookup | `lookup "v1" "Secret" .Release.Namespace "x"` | `lookup("v1", "Secret", release.namespace, "x")` |

</details>

## Le nouveau modèle de `lookup()`

Le `lookup` de Helm interroge le cluster au moment du rendu pendant
`install`/`upgrade` et retourne `{}` pendant `helm template`. Sherpack
fait **exactement la même chose**, mais avec un twist : chaque résultat
non-vide produit un warning structuré pour que le non-déterminisme
ne soit pas silencieux.

## Pièges à connaître

<div className="migration-gotcha-list">

  <details className="migration-gotcha">
    <summary>
      <span className="migration-gotcha-icon" aria-hidden="true">⚠</span>
      Les warnings <code>lookup()</code> demandent un niveau de log élevé
    </summary>
    <p>
      Sherpack utilise <code>tracing</code>. La CLI est en niveau{' '}
      <code>warn</code> par défaut sur les crates sherpack et écrit sur
      stderr — tu vois donc les warnings lookup. Pour plus de détail,
      mets <code>RUST_LOG=info</code> ou utilise <code>--debug</code>.
    </p>
  </details>

  <details className="migration-gotcha">
    <summary>
      <span className="migration-gotcha-icon" aria-hidden="true">⚠</span>
      <code>.helmignore</code> n'est pas honoré au packaging
    </summary>
    <p>
      Le converter le renomme en <code>.sherpackignore</code> mais{' '}
      <code>sherpack package</code> ne le lit pas encore. Si tu as des
      fichiers à exclure du paquet, supprime-les avant de packager.
    </p>
  </details>

  <details className="migration-gotcha">
    <summary>
      <span className="migration-gotcha-icon" aria-hidden="true">⚠</span>
      Certaines fonctions Sprig n'ont pas d'équivalent direct
    </summary>
    <p>
      <code>dateModify</code>, <code>dateInZone</code>, <code>htpasswd</code>,
      <code>bcrypt</code>, <code>fromToml</code>/<code>toToml</code>, et
      les helpers OS/URL ne sont pas portés. Le converter les flag pour
      qu'ils apparaissent dans le rapport de warnings.
    </p>
  </details>

  <details className="migration-gotcha">
    <summary>
      <span className="migration-gotcha-icon" aria-hidden="true">⚠</span>
      Pas (encore) d'écosystème de plugins
    </summary>
    <p>
      Des outils comme helm-diff, helm-secrets, helm-git n'ont pas de
      remplaçants directs. Le natif Sherpack couvre le diff
      (<code>--diff</code>), la génération idempotente de secrets, et les
      lock files — mais si tu dépends fortement de plugins, évalue
      avec attention.
    </p>
  </details>

  <details className="migration-gotcha">
    <summary>
      <span className="migration-gotcha-icon" aria-hidden="true">✓</span>
      Tu peux faire tourner Sherpack et Helm côte à côte
    </summary>
    <p>
      Sherpack stocke les releases sous un label différent
      (<code>app.kubernetes.io/managed-by=sherpack</code>) et utilise des
      annotations différentes. Les releases Helm restent invisibles à
      Sherpack et inversement — pratique pendant une migration progressive.
    </p>
  </details>

## FAQ

<details className="migration-faq">
<summary>Mes charts Helm existants continueront-ils de fonctionner avec Helm ?</summary>
<p>
Oui. <code>sherpack convert</code> écrit un nouveau Pack dans un dossier
séparé ; ton chart d'origine n'est pas touché. Fais tourner les deux côte
à côte tant que tu n'es pas confiant dans la migration.
</p>
</details>

<details className="migration-faq">
<summary>Et les subcharts et les umbrella charts ?</summary>
<p>
Sherpack gère les subcharts avec la même sémantique de scoping de values
et de globals que Helm. Le dossier <code>charts/</code> devient{' '}
<code>packs/</code> ; les références dans <code>Pack.yaml</code> miroitent
le bloc dependency du <code>Chart.yaml</code> de Helm.
</p>
</details>

<details className="migration-faq">
<summary>La sortie rendue est-elle byte-identique à celle de Helm ?</summary>
<p>
Pour la plupart des charts, oui — modulo les whitespaces et les
différences listées dans l'antisèche. Lance{' '}
<code>sherpack template</code> et <code>helm template</code> sur les mêmes
entrées et fais un diff pour vérifier avant d'installer en production.
</p>
</details>

<details className="migration-faq">
<summary>Puis-je utiliser Sherpack avec ArgoCD ou Flux ?</summary>
<p>
Oui — rends avec <code>sherpack template</code> et laisse le contrôleur

<details className="migration-faq">
<summary>Performance vs Helm ?</summary>
<p>
Binaire Rust natif, ~19 Mo, pas de runtime Go. Les temps de rendu pour
les charts typiques sont sub-secondes. Le moteur MiniJinja est rapide,
mais les vrais gains sont sur la recherche (SQLite FTS5) et la
résolution de dépendances.
</p>
</details>
